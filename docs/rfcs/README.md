# Request for Comments (RFC)

新機能・大きな仕様変更の**提案と実装計画**を置く場所。

## RFC とは

「これから作るもの」を、実装に入る前に文章で固める。Rust RFC / Python PEP と同じ発想。目的は 3 つ:

1. **Spec を明文化する**: 何を作るか、どう振る舞うかを書き出す（superpowers の `/brainstorming` の成果）
2. **実装計画を共有する**: どう作るか、どの順で進めるかを書き出す（superpowers の `/writing-plans` の成果）
3. **判断の記録にする**: なぜその設計にしたか、検討した代替案を残す

実装が終わっても**消さない**。Status を `Implemented` にして履歴として残す。

## 何を RFC にするか

- 新しいエンドポイントや UI ページの追加
- 既存機能の挙動を大きく変える変更（例: 会話履歴の永続化、認証導入）
- 複数モジュールに波及する設計変更
- バックログの大きめなアイデア（リトライ、シナリオ拡張、SRS 改良など）

逆に、バグ修正・小さなリファクタ・依存追加程度は RFC にしない（コミットメッセージと PR で十分）。

## ADR との違い

| | RFC | ADR |
| --- | --- | --- |
| 視点 | 前向き（これから作る） | 後ろ向き（決まったことを記録） |
| 内容 | Spec + 実装計画 | 設計判断 + 代替案 + 不採用理由 |
| 起こすタイミング | 機能を提案する時 | 設計判断が固まった時 |
| 関係 | RFC が Accepted → 重要な設計判断は ADR に落とす | RFC を元に書かれることが多い |

両方を一気に書く必要はない。小さい RFC なら ADR が要らないこともあるし、ADR だけで済む判断（ライブラリ選定など）は RFC を経由しなくていい。

## 書き方

1. [`template.md`](./template.md) をコピーして `NNNN-<slug>.md` を作る（連番、欠番なし）。
2. Status を `Draft` で開始。書き終えたら `In Review` に上げてレビューを受ける。
3. Accepted になったら実装に入る（TDD で進める）。実装完了後は `Implemented` に更新。
4. 重要な設計判断が含まれる場合は、Accepted のタイミングで ADR を起こして相互リンクする。
5. 過去の RFC を**書き換えない**。方針変更は新しい RFC を起こし、古い方は Status を `Superseded by NNNN` に更新する。

## Status の種類

- `Draft` — 執筆中。まだ議論の俎上に載っていない
- `In Review` — レビュー依頼中。Spec / Plan に対するフィードバックを集める
- `Accepted` — 実装着手 OK。これから TDD で進める
- `Implemented` — 実装完了。コードがマージ済み
- `Rejected` — 採用しないと決定。理由を本文に追記
- `Superseded by NNNN-<slug>` — 別の RFC に置き換わった
